================================================================================
 Decode Benchmark — Expanded Commands

 What launch_bench.sh actually runs on each node, with all variables resolved.
 Configuration: 4 nodes, EP=32, deepep_low_latency, GLM-5-FP8
================================================================================

Nodes (one per line in /tmp/bench_nodes.txt):
  node0 = HEAD       (e.g. ltc-idc3-hgx8-h200-12,  IP = 10.20.0.18)
  node1 = worker 1   (e.g. ltc-idc3-hgx8-h200-35)
  node2 = worker 2   (e.g. ltc-idc3-hgx8-h200-36)
  node3 = worker 3   (e.g. ltc-idc3-hgx8-h200-37)

================================================================================

# Run this on EACH node before launching:

pids=$(pgrep -f "vllm serve|vllm.entrypoints|EngineCore|ApiServer|DPCoordinator|multiprocessing.resource_tracker" 2>/dev/null | grep -v $$)
if [ -n "$pids" ]; then echo "$pids" | xargs kill -9 2>/dev/null; fi

================================================================================
 WHAT CHANGES BETWEEN NODES
================================================================================

Only three things differ per worker:

  --data-parallel-start-rank  N    (8, 16, 24 for workers 1, 2, 3)
  --data-parallel-address     IP   (always the HEAD node's IP)
  --headless                       (present on workers, absent on head)

Head has instead:
  --host 0.0.0.0
  --port 8200
  --data-parallel-address $D_IP    (its own IP)

Everything else is identical across all 4 nodes.


================================================================================
 WHAT EACH FLAG DOES
================================================================================

--data-parallel-size 32         Total EP rank count across all nodes (4 × 8 GPUs)
--data-parallel-size-local 8    EP ranks per node (1 per GPU)
--data-parallel-start-rank N    First EP rank on this node (head=0, w1=8, w2=16, w3=24)
--data-parallel-address IP      Head node IP for ZMQ handshake + Gloo rendezvous
--data-parallel-rpc-port 29560  ZMQ ROUTER port on head for startup handshake
--enable-expert-parallel        Use Expert Parallelism for MoE layers
--all2all-backend               deepep_low_latency = NVSHMEM A2A (prod)
                                deepep_high_throughput = NCCL A2A (simpler)
--headless                      Worker mode: no HTTP server, EngineCore only
--kv-transfer-config            DecodeBenchConnector fills KV with random values
--compilation-config            FULL_DECODE_ONLY = full CUDA graphs for decode only


================================================================================
 ENVIRONMENT VARIABLES
================================================================================

NCCL_NET_PLUGIN=none          Prevents NCCL from loading HPC-X UCX plugin
                              (which would steal UCM hooks from NIXL's UCX)
GLOO_SOCKET_IFNAME=bond0      torch.distributed gloo uses bond0 (not eth0)
DG_JIT_CACHE_DIR=/tmp/...     Local DeepGEMM JIT cache (avoids NFS Stale file handle)
HF_HUB_OFFLINE=1              Don't call HuggingFace Hub (model is cached)
HF_HOME=/shared/huggingface   Shared model cache path
LD_LIBRARY_PATH               Must include NVSHMEM lib dir for deep_ep
TOKENIZERS_PARALLELISM=false   Suppress HuggingFace tokenizer fork warning


================================================================================
 7. BENCHMARKING WITH GUIDELLM (recommended over vllm bench serve)
================================================================================

# Install (once):
pip install 'guidellm[recommended]'

# Throughput benchmark — runs for 120s at max throughput, measures steady state:
guidellm benchmark run \
    --target "http://10.20.0.18:8200" \
    --data "prompt_tokens=1024,output_tokens=1024" \
    --profile throughput \
    --max-seconds 120 \
    --request-type text_completions \
    --processor-args '{"trust_remote_code": true}' \
    --backend-kwargs '{"timeout": 600}' \
    --output-path results/guidellm/throughput_in1024_out1024.json

# Concurrent sweep — test specific concurrency levels:
guidellm benchmark run \
    --target "http://10.20.0.18:8200" \
    --data "prompt_tokens=1024,output_tokens=1024" \
    --profile concurrent \
    --rate "32,64,128,256" \
    --max-seconds 120 \
    --request-type text_completions \
    --processor-args '{"trust_remote_code": true}' \
    --backend-kwargs '{"timeout": 600}' \
    --output-path results/guidellm/concurrent_sweep.json

# Single concurrency level, fixed request count:
guidellm benchmark run \
    --target "http://10.20.0.18:8200" \
    --data "prompt_tokens=1024,output_tokens=1024" \
    --profile concurrent \
    --rate 64 \
    --max-requests 256 \
    --request-type text_completions \
    --processor-args '{"trust_remote_code": true}' \
    --backend-kwargs '{"timeout": 600}' \
    --output-path results/guidellm/bench_c64.json


IMPORTANT: --backend-kwargs '{"timeout": 600}' is REQUIRED.
Default is 60s which is too short for 1024-output-token decode.
At EP=32, straggler requests (last few in a batch) run slowly because
all 32 engines synchronize on every forward step — when most engines
have finished their requests, the remaining 1-2 still running cause
all engines to idle-wait. This can push individual request latency to
60-120s+ at high concurrency. 600s timeout avoids false errors.

PROFILES:
  throughput   — max requests/s, no concurrency limit (best for peak throughput)
  concurrent   — fixed number of concurrent requests (best for latency at load)
  sweep        — auto-discover saturation point (default, good for exploration)
  synchronous  — one request at a time (baseline latency)
  constant     — fixed requests per second
  poisson      — poisson-distributed arrivals
